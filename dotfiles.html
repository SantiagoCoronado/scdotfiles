What is it?

A dotfiles repository is a git repo that stores all your personal configuration files — shell settings, editor preferences, terminal themes, tool configs — in one version-controlled place. This one adds automated bootstrap scripts so you can go from a bare machine to a fully configured dev environment in minutes.

The Core Idea: Symlinks

Instead of copying config files to where apps expect them, install.sh creates symbolic links. The file lives in the git repo; the symlink makes the app think it's in the usual place.

# What actually exists on disk:
~/VSCodeProjects/dotfiles/shared/config// ← real files, in git

# What Neovim sees:
~/.config/ → ~/VSCodeProjects/dotfiles/shared/config//

# Edit from either path — same file:
~/.config//init.lua # you're editing the git-tracked file

Symlinks vs. Copying

Edit a configInstantly in git ✓Must remember to copy back
Sync machinesgit pull + ./install.sh ✓Manual re-copy everywhere

HistoryFull git log ✓None
ComplexityOne-time setup ✓Simple but fragile

Purpose

Every developer eventually faces the same pain: a new machine, a fresh VPS, or a WSL2 instance — and hours of manual setup just to feel at home. This project solves that.

🖥️

Identical Environments

Your Neovim, tmux, shell, and git behave exactly the same on your MacBook, a remote VPS, or inside WSL2. No more "it works on my machine."

⚡

Zero-to-Productive Fast

Bootstrap scripts automate every step: package install, Oh My Zsh, plugins, dotfiles clone, symlinks, LSP servers. A VPS is ready in one command.

📦

Version-Controlled History

Every config change is a commit. You can see exactly when you changed a tmux keybind, diff your Neovim setup over time, and roll back anything.

🔀

Platform-Aware

macOS-only tools (sketchybar, karabiner) never pollute Linux. Shared configs work everywhere or degrade gracefully. The split is enforced by the directory structure.

🔒

Secrets Stay Secret

A strict .gitignore blocks .env files, OAuth tokens (gh/hosts.yml), karabiner backups, and all Claude Code runtime data. Public repo-safe by design.

🤖

AI-Integrated Workflow

The claude/ directory ships a full Claude Code setup: custom agents, slash commands, hooks, and rules — version-controlled alongside your other tools.

What install.sh actually does

It's 84 lines of bash that does exactly one thing: create symlinks. Here's the logic:

mkdir -p Creates ~/.config/{git,npm,gh,btop,lazygit,thefuck} and ~/.claude/ if they don't exist
ln -sf File-level symlinks for single config files (git/config, npmrc, btop.conf…)
rm + ln -sf Directory symlinks: removes old link first, then links entire dirs (, tmux, claude/agents…)
--macos Also links sketchybar and karabiner (macOS-exclusive tools)
idempotent Safe to run multiple times — set -e on errors, -f flag on ln overwrites old symlinks cleanly

Day-to-day sync workflow

# Edit a config on machine A:
~/.config//init.lua # editing git-tracked file directly
cd ~/VSCodeProjects/dotfiles
git add -p && git commit -m ": tweak autocmd" && git push

# Sync to machine B:
git pull # symlinks already point at the right place — nothing else needed

Should You Use It?

✓

You should use it if…

●
You work across multiple machines. MacBook for dev, VPS for deployments, WSL2 at work? One repo, one command, identical environments.
●
You want to version-control your config. See exactly when you changed a keybind. Roll back a broken Neovim config. Diff your shell setup over 2 years.
●
You spin up new machines regularly. A fresh Arch VPS or WSL2 instance goes from bare to fully configured in under 10 minutes, fully unattended.
●
You use (or want to use) Neovim heavily. The config alone is worth stealing — full LSP, 25+ plugins, language-specific ftplugins, all in organized Lua.
●
You use Claude Code. The claude/ directory ships a production-ready setup: custom agents, slash commands, hooks with TTS, security rules.
●
You want a curated modern toolchain. Every tool was chosen deliberately: bat over cat, eza over ls, ripgrep over grep, lazygit over CLI git, delta for diffs.
✗

You shouldn't use it if…

●
You're a zsh/bash beginner. The bootstrap scripts run with root-level side effects on your machine. Understanding what they do before running them is essential.
●
You want a GUI-first setup. Everything here is terminal-centric. VS Code with a GUI is not part of this setup. This is built around Neovim, tmux, and the CLI.
●
You don't use macOS, Arch, or Ubuntu/Debian. Bootstrap scripts use pacman and apt. Other distros (Fedora, openSUSE) would need modification.
●
You have different tool preferences. This is an opinionated setup. If you prefer fish over zsh, Helix over Neovim, or Starship alternatives — you'll spend more time adapting than it's worth.
●
You don't have SSH access to a different GitHub host. bootstrap-vps.sh clones from git@github-mourey:… (a custom SSH host alias). You'd need to update this to your own repo URL.
●
You're looking for a "plug and play" community solution. This is a personal repo with personal opinions. It's better used as inspiration or a starting point than run verbatim on your machine.

Honest verdict

This is a well-structured personal dotfiles repo that solves real problems. The symlink approach is solid, the platform separation is clean, and the bootstrap scripts are thorough. Even if you don't use it directly, it's excellent reference material for building your own. The Claude Code integration in particular is unusually complete — most dotfiles repos don't treat AI tooling as a first-class citizen.

Quick Start

1. Install prerequisites 
# Install Homebrew
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# Install tools
brew install neovim tmux fzf zoxide bat eza ripgrep fd git-delta btop lazygit gh atuin mise direnv thefuck starship
brew install --cask ghostty wezterm
2. Clone & install 
git clone https://github.com/Mourey/dotfiles ~/VSCodeProjects/dotfiles
cd ~/VSCodeProjects/dotfiles
./install.sh --macos
Run from your local machine 
# Clone dotfiles locally first
git clone https://github.com/Mourey/dotfiles ~/VSCodeProjects/dotfiles
cd ~/VSCodeProjects/dotfiles

# Bootstrap a bare Arch Linux VPS (replace 'my-vps' with your SSH host)
./bootstrap-vps.sh my-vps

Requires: SSH access configured. Script SCPs itself to the VPS and runs remotely.

Run inside the WSL2 instance 
# Clone dotfiles
git clone https://github.com/Mourey/dotfiles ~/VSCodeProjects/dotfiles
cd ~/VSCodeProjects/dotfiles

# Run the WSL2 bootstrap (Ubuntu/Debian)
./bootstrap-wsl.sh

Installs all packages, sets up zsh, Oh My Zsh, tools, win32yank, and runs install.sh --linux automatically.

Keeping configs in sync (any platform) 
cd ~/VSCodeProjects/dotfiles
git pull          # Fetch latest changes
./install.sh --macos   # Re-run if symlink targets changed (idempotent)
